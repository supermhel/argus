# FENGARDE v0.4 — Build Plan (auth, niche parser packs, reporting hook, distribution)

**Date:** 2026-07-10
**Baseline:** v0.3 Tracks A/B/C1 landed on `main` (unreleased — see Track 0). 7 parsers, 8 rules,
enrichment substrate, triage workflow with OCC, backpressure, two rule CI gates. See `SSOT.md` for
current state and `2026-07-02-fengarde-v0.3-improvement-plan.md` (now annotated) for what carried over.
**Strategy input:** July-2026 market analysis (ecosystem/niche map, FENGARDE market deep-dive, SigNoz
playbook study — user-provided). Validated positioning:

> **"The open-source SIEM for the European industrial Mittelstand — turns your factory and IT logs
> into NIS2/DORA evidence, with AI triage that never leaves your network."**

**Goal of v0.4:** make FENGARDE *adoptable and positioned* — real (minimal, honest) auth so anyone can
run it beyond localhost; the three niche parser packs the analysis identified (MCP/agent logs,
OT/OPC UA, n8n); the open half of the NIS2/DORA incident-report feature (hook + generic template —
regulatory content lives in fengarde-sec, see cross-repo contract below); and the homelab quickstart +
content that starts the community flywheel. No architecture rewrites: everything is additive to the
verified keystone.

**Locked decisions feeding this plan (2026-07-10):**
1. Report generator **split**: hook + generic template open (this repo); legally-gated German
   NIS2/DORA regulatory content paid (fengarde-sec). Respects fengarde-sec's C3 additive-field constraint.
2. fengarde-sec goes **RAG-first, GPU training deferred** until a design partner exists — so this
   repo's reporting hook is the near-term integration surface, not the WS-5 model swap.
3. **All three parsers** (MCP, OT, n8n) in v0.4 scope.

Each item: **why**, **what**, **files**, **effort** (human / with-CC), **risk**, **priority**.

## Track S — Auth & hardening (the adoption prerequisite)

The market analysis is blunt: "auth + hardening on FENGARDE so anyone can actually run it beyond
localhost (prerequisite for everything)." Today no service has any authentication (SECURITY.md §2).
v0.4 ships the *minimal honest* version: opt-in shared-secret auth on the HTTP write/read surfaces
and the bus, not a full identity system. Full authN/authZ (users, roles, TLS) stays out of scope
and documented.

**Design principle for the whole track:** auth is **enabled by setting an env var, disabled with a
startup warning when unset**. Zero-infra tests and the quickstart keep working unchanged; a real
deployment sets the vars. SECURITY.md states this trade-off explicitly — default-off auth is a
documented choice, not an oversight.

### S1. API-key middleware for the HTTP APIs (WS-6 inventory, WS-3 triage)
**Why:** these are the two unauthenticated *write* surfaces (SECURITY.md §7 documents the triage
API's blast radius). One shared header check covers both.
**What:** `services/shared/authz.py` — a single helper `require_api_key(handler) -> bool` used by
both stdlib `http.server` handlers: when `FENGARDE_API_KEY` is set, every request must carry
`X-Api-Key: <value>` (constant-time compare via `hmac.compare_digest`); mismatch/missing → 401 JSON
`{"error": "unauthorized"}`. When unset → allow all + one startup warning log line
(`"auth disabled: FENGARDE_API_KEY not set"`). Wire into `services/ws6-inventory/app.py` and
`services/ws3-indexer/triage_api.py` (before routing, after path parse). Dashboard's nginx
`/api/triage` proxy passes the header through (`proxy_set_header X-Api-Key ...` from an env-templated
conf, or the dashboard JS sends it — decide at build time; simplest: nginx injects it so the browser
never holds the key).
**Files:** `services/shared/authz.py` (new), `services/ws6-inventory/app.py`,
`services/ws3-indexer/triage_api.py`, `services/ws3-indexer/test_auth.py` (new),
`services/ws6-inventory/test_auth.py` (new), `infra/docker-compose.yml` (env plumbing),
dashboard nginx conf.
**Effort:** 1d / 0.5d. **Risk:** low — but the middleware must run before ANY route logic (no
authenticated-path confusion), and tests must cover both modes (unset = open + warning; set = 401
on missing/wrong, 200 on correct). **Priority: P0.**

### S2. Dashboard basic-auth (opt-in, at nginx)
**Why:** the dashboard (:8080) is the human read surface; a homelab user exposing it on a LAN wants
at least a password.
**What:** nginx `auth_basic` with an htpasswd file mounted via compose; enabled when
`DASHBOARD_HTPASSWD` (path) is set, absent otherwise. Document `htpasswd -c` one-liner in README.
**Files:** `services/ws7-dashboard/nginx.conf` (or its template), `infra/docker-compose.yml`.
**Effort:** 2h / 1h. **Risk:** none. **Priority: P1.**

### S3. Redis AUTH
**Why:** the bus (:6379) is the spine; anyone reaching it can inject/consume every topic. Redis
`requirepass` is one env var of protection.
**What:** compose sets `--requirepass ${REDIS_PASSWORD}` when the var is set; `services/shared/bus.py`
RedisBus passes `password=os.getenv("REDIS_PASSWORD")` (None = unchanged). Memory backend untouched.
The CI `redis-integration` job runs once with a password set to prove the path.
**Files:** `services/shared/bus.py`, `infra/docker-compose.yml`, `.github/workflows/ci.yml`.
**Effort:** 2h / 1h. **Risk:** low. **Priority: P1.**

### S4. OpenSearch security plugin — explicitly stays disabled in v0.4
**Why:** enabling it means TLS certs, users, and roles across every service that touches :9200 —
disproportionate to v0.4's goal and it would break the zero-friction local story.
**What:** a *documentation* decision: OpenSearch stays `DISABLE_SECURITY_PLUGIN=true`, mitigation is
"don't publish :9200/:5601 beyond localhost" (already the compose default — verify and pin it:
ports bound to `127.0.0.1` in compose, not `0.0.0.0`). SECURITY.md gets a v0.4 threat-boundary
update describing exactly what S1–S3 do and do not protect (no TLS anywhere, shared static key ≠
identity, OpenSearch open on localhost).
**Files:** `SECURITY.md`, `infra/docker-compose.yml` (bind addresses).
**Effort:** 2h. **Risk:** none (honesty is the deliverable). **Priority: P1.**

## Track P — Niche parser packs (the three wedges)

Each parser follows the worn pattern (`docs/adding-a-parser.md`): copy `linux_ssh.py` structure →
`services/ws2-normalization/parsers/<name>.py`, register in `parsers/__init__.py`, fixture in
`fixtures/`, sample in `test_contract.py`, rules in `contracts/rules/*.yml`. Every rule must pass
both CI gates (`tools/validate_rules.py` schema/grammar gate + `tools/check_rule_producers.py`
anti-dormancy proof). All parsers process untrusted input — the established ReDoS/DoS discipline
applies (no catastrophic regex, size caps, fail-closed on malformed lines).

### P1. MCP / AI-agent audit-log parser + rule pack (the attention play)
**Why:** 88% of orgs report AI-agent security incidents; nobody ships open-source detection content
for agent telemetry. One parser + three rules = the highest-visibility, lowest-cost item in the
plan (HN/r/netsec material). EU AI Act high-risk logging obligations bite 2026-08-02.
**What:**
- **Input format:** JSONL lines as emitted by MCP server/gateway logs — JSON-RPC `tools/call`
  records: `{ts, session_id, agent, server, tool, arguments, outcome}`. Define the exact accepted
  shape in the parser docstring + fixture; tolerate vendor variance via a small field-alias map
  (this is a *log* parser, not an MCP client).
- **OCSF mapping:** class **6003 API Activity** — `api.operation` = tool name,
  `actor.user.name` = agent/session identity, `api.request.data` = arguments JSON truncated to a
  hard cap (attacker-controlled — cap bytes, never parse-execute), `src_endpoint` from client
  info when present, `unmapped.mcp_server`, `unmapped.session_id`. `type_uid` derived, never
  hand-set.
- **Rules (`contracts/rules/`):**
  - `agent_credential_file_access.yml` — single-shot: tool arguments reference a
    secret-material path (`.env`, `id_rsa`, `.aws/credentials`, `secrets.*`, etc.). Substring/
    pattern list documented in the rule as a heuristic.
  - `agent_tool_call_burst.yml` — windowed count per `session_id` above threshold (runaway/abused
    agent).
  - `agent_prompt_injection_indicator.yml` — single-shot: known injection markers in `api.request.data`
    ("ignore previous instructions", "system prompt", exfil-URL patterns). **Labeled honestly in
    the rule description as a string-match heuristic, not a classifier** — no ML claim.
**Files:** `services/ws2-normalization/parsers/mcp_agent.py` (+ registration + tests),
`fixtures/mcp_agent.jsonl`, 3 rule files, `contracts/detection-coverage.md` row.
**Effort:** 1.5d / 1d. **Risk:** low technically; medium on format choice — there is no single
standard MCP log format yet, so the parser defines one and documents it (that's a feature: "here's
the shape to log in" is itself content). **Priority: P1.**

### P2. OT parser — OPC UA audit events first (the moat)
**Why:** OT incumbents are enterprise-priced; Wazuh is endpoint-first; an OCSF-normalizing OT parser
pack is near-unique in open source, and it's the core of the industrial-Mittelstand wedge (NIS2
covers manufacturing). Founder background (Mechatronik, Siemens industrial monitoring) is the
credibility asset here.
**What — why OPC UA and not S7/PROFINET first:** OPC UA has a *publicly documented, structured
audit event model* (Part 5: `AuditCreateSessionEventType`, `AuditActivateSessionEventType`,
`AuditWriteUpdateEventType`, `AuditCertificateEventType`, method-call audit events); S7/PROFINET
telemetry is proprietary-shaped and needs hardware access to fixture honestly. S7 is deferred to
v0.5 — stated in the coverage map, not silently dropped.
- **Input format:** OPC UA server audit log lines (JSON export of audit events — document the
  accepted shape; fixture from the OPC UA spec's event fields).
- **OCSF mapping:** session/activate/certificate audit events → **3002 Authentication**
  (`actor.user.name` = client identity/cert CN, `src_endpoint` = client address); write/method-call
  audit events → **6003 API Activity** (`api.operation` = "Write"/method name,
  `unmapped.ot_node_id` = target NodeId, `device.type = "plc"`).
- **Rules:**
  - `ot_write_outside_maintenance.yml` — write/config change outside a configured maintenance
    window (**reuses the existing `outside_hours` predicate** — zero grammar work).
  - `ot_new_engineering_connection.yml` — windowed distinct-count: sessions to one PLC/server from
    a source not seen in the window baseline (new engineering workstation).
  - `ot_config_change.yml` — single-shot on write to configuration/firmware NodeIds (pattern list
    in rule).
**Files:** `services/ws2-normalization/parsers/opcua_audit.py` (+ registration + tests),
`fixtures/opcua_audit.jsonl`, 3 rule files, coverage-map row.
**Effort:** 1.5d / 1d. **Risk:** medium — fixture realism: no live PLC to record from, so fixtures
are spec-derived; label them as such (honest-status rule). The parser ships as "OPC UA audit-log
format as specified in Part 5" — a design partner with a real Siemens/B&R stack validates later.
**Priority: P1.**

### P3. n8n / automation-platform parser + rule pack (opportunistic niche)
**Why:** 100k+ internet-facing n8n instances, governance features paywalled at €667+/mo, self-hosters
have nothing watching the platform. Small cost, real niche, and founder knows n8n from own
automation work.
**What:**
- **Input:** n8n audit/execution JSON log events (workflow created/updated/activated, webhook
  created, credential accessed, execution start/finish, login).
- **OCSF:** workflow/webhook/credential ops → **6003 API Activity**; login events → **3002
  Authentication**. `actor.user.name` = n8n user, `api.operation` = event type,
  `unmapped.workflow_id`, `unmapped.webhook_path`.
- **Rules:**
  - `n8n_new_webhook_exposed.yml` — single-shot on webhook-created (new unauthenticated entry
    point).
  - `n8n_workflow_modified_after_hours.yml` — workflow update outside business hours (**reuses
    `outside_hours`**).
  - `n8n_credential_access_burst.yml` — windowed count of credential-node accesses per user.
**Files:** `services/ws2-normalization/parsers/n8n_audit.py` (+ registration + tests),
`fixtures/n8n_audit.jsonl`, 3 rule files, coverage-map row.
**Effort:** 1d / 0.5d. **Risk:** low. **Priority: P2** (after P1/P2 — same pattern, less strategic
weight).

### P4. Impossible-travel rule (v0.3 A2 leftover — now unblocked by A5 geo enrichment)
**Why:** the geo substrate (`src_endpoint.location.country` from `contracts/enrichment/geoip.yml`)
shipped in v0.3 with zero consumers. This is the first rule that pays for it, and it's a classic
credential-theft detection.
**What:** `common_impossible_travel.yml` — same account, successful auth (`3002`, activity 1) from
≥2 **distinct `src_endpoint.location.country` values** within a short window (simplified
impossible-travel: country-change, not velocity math — honest labeling in the rule description).
Mechanically this is the existing distinct-count window (`group_by` user, `distinct_field`
`src_endpoint.location.country`, threshold 2). **Engine check:** confirm the distinct-count path
resolves dotted nested fields on enriched events; if not, minimal extension covered by
`test_window_distinct.py`. Producer proof: extend `contracts/enrichment/geoip.yml` sample entries +
linux_ssh/windows fixtures so `check_rule_producers.py` proves satisfiability.
**Files:** `contracts/rules/common_impossible_travel.yml`, possibly
`services/ws4-detection/window.py` (nested-field distinct), `test_engine_distinct_rules.py` case,
fixture/geoip sample touch-ups.
**Effort:** 0.5d / 2h. **Risk:** low; watch the acceptance-e2e alert counts (adding rules perturbed
them before — update expected counts deliberately). **Priority: P1.**

## Track R — Incident-report hook (the open half of the killer feature)

The analysis' sharpest finding: *"raw logs → correlated alert → draft regulator-ready incident
report. No open-source tool does this."* The split decision puts the **hook + a generic template**
here (open, demoable, community-visible) and the **regulatory content** (German NIS2/DORA field
mappings, RAG over verbatim regulation text, deadline logic, citations) in fengarde-sec as the paid
backend. This track is also fengarde-sec's C3 "additive open-core field" constraint made concrete: the
open pipeline is complete and useful with the builtin template backend; the paid backend only makes
the *content* better.

### R1. Reporting contract (`contracts/reporting.md`) — freeze the seam first
**Why:** two repos build against this; the contract must exist before either side codes. Same
discipline as `contracts/bus-topics.md`.
**What:** a new frozen contract file defining:
- **Trigger:** `POST /alerts/{alert_id}/report` on the existing WS-3 triage API port (8013). WS-3
  owns alerts; no new service. Calling an external report backend over HTTP is the same pattern as
  WS-5 → Ollama (external callee, not inter-WS coupling — bus-only coupling between workstreams is
  untouched).
- **Backend seam:** `REPORT_BACKEND=template` (default, builtin) | `http` (POST the request payload
  to `FENGARDE_SEC_REPORT_URL`, e.g. the fengarde-sec report-draft service). Backend selection mirrors
  `llm_adapter.py::make_llm()`'s env-driven pattern. HTTP backend failure → fall back to the
  template backend + a `backend_degraded` field (fail-open, report still produced).
- **Request payload (fengarde → backend):**
  `{alert: <full alert doc>, triage: <triage state>, events: [<up to REPORT_MAX_EVENTS contributing
  events>], requested_at}`.
- **Response schema (frozen):**
  ```json
  {
    "report_id": "<alert_id>:report",
    "alert_id": "...",
    "format": "markdown",
    "body": "...",
    "status": "draft",
    "disclaimer": "DRAFT — automatically generated. Not legal advice. Review before any regulatory submission.",
    "generated_at": 0,
    "backend": "template | fengarde-sec",
    "citations": [{"celex": "...", "article": "...", "url": "...", "retrieved_at": "..."}]
  }
  ```
- **Hard rules (written into the contract):** (1) `status` enum contains **only `"draft"`** in
  v0.4 — the enum widens only when fengarde-sec's legal sign-off gate passes; (2) `disclaimer` is
  mandatory and non-empty; a response without it is rejected and the builtin template used instead;
  (3) `citations` is optional and may be empty — the open pipeline never depends on it (tolerant
  reader; this IS the additive-field constraint).
- **Storage:** report doc indexed as `reports-YYYY.MM.DD` via the existing storage adapter
  (`report_id` deterministic = idempotent under retry, same discipline as alerts).
  `GET /alerts/{alert_id}/report` returns the stored report or 404.
**Files:** `contracts/reporting.md` (new).
**Effort:** 0.5d / 2h. **Risk:** low — but this is a *frozen contract*; get the field names right
once. **Priority: P0 for the track.**

### R2. Builtin template backend + API + dashboard button
**Why:** the open, demoable half — an alert becomes a structured incident report in one click,
with zero paid dependency.
**What:**
- `services/ws3-indexer/reporting.py` (new): request assembly (alert + triage + contributing
  events from the store), backend dispatch per R1, response validation (schema + the two hard
  rules), storage.
- **Template backend:** renders generic incident-report markdown from alert facts — rule
  title/severity/score, detection timeline (first/last event ts), source/target endpoints,
  observables (IPs/users/hosts from the events), current triage state, and an "unfilled fields"
  section (what a human must add). **Zero regulatory claims** — no NIS2/DORA article references in
  the open template (that content is fengarde-sec's asset; a generic report is the community
  feature).
- API routes on the triage server (`triage_api.py` dispatch): `POST/GET /alerts/{id}/report`, same
  auth (S1), body caps, malformed-input discipline as the triage endpoints.
- Dashboard: "Draft report" button on the alert row/triage view → calls POST via the same nginx
  path, renders markdown body in a panel; shows the disclaimer prominently.
- Tests (`services/ws3-indexer/test_reporting.py`, new): schema validation, template rendering
  from a fixture alert, backend seam with a fake HTTP callee (success / non-conforming response
  rejected / timeout falls back to template), idempotent re-generation, auth applied.
**Files:** as above + `run_all_tests.sh` wiring + `contracts/opensearch mappings` addition for the
reports index.
**Effort:** 2d / 1d. **Risk:** medium — response validation must be strict (the backend is
external input); markdown renders into the dashboard, so the existing XSS discipline applies
(render as text/sanitized, never innerHTML). **Priority: P1.**

## Sequencing (what to actually do, in order)

1. **0.1** tag v0.3.0 (15 min, unblocks clean v0.4 history).
2. **S1** API-key middleware → **S3** Redis AUTH → **S2** dashboard basic-auth → **S4** SECURITY.md
   v0.4 boundary (auth track complete before anything is publicized).
3. **R1** reporting contract (freeze the seam — fengarde-sec's Track R builds against it in parallel
   from here).
4. **R2** template backend + API + dashboard button.
5. **P1** MCP parser + rules (the attention play — ready before launch).
6. **D1** quickstart (`make demo`) → **D2** README wedge.
7. **P2** OPC UA parser + rules → **P4** impossible-travel → **P3** n8n parser + rules.
8. **D3** content drafts + launch checklist → *human action: post*.

Rough total: ~10–12 with-CC days of work.

**Golden rules (unchanged + one new):** bus-only coupling; OCSF-additive fields with tolerant
readers; stateless workers; RCE-free fail-closed evaluator; deterministic IDs / idempotency under
at-least-once delivery; zero-infra gate green before "done"; honest status — no stubbed capability
labeled working. **New for v0.4:** every generated report is `status: "draft"` with a mandatory
disclaimer until fengarde-sec's legal sign-off gate passes — this repo enforces it structurally
(schema validation rejects anything else), not by convention.
