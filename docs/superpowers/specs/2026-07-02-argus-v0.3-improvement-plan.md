# ARGUS v0.3 — Improvement Plan (detection, logic, architecture, design)

**Date:** 2026-07-02
**Baseline:** v0.2.0 (6 parsers, 5 rules, live-proven pipeline). See `SSOT.md` for current state
and `2026-07-02-argus-architecture-review.md` for the structural findings this plan acts on.
**Goal of v0.3:** make ARGUS materially *better at being a SIEM* — more detection coverage, more
robust rule logic, the architecture fix that unblocks scale, and a triage-first dashboard — without
disturbing the verified keystone (bus-only coupling, OCSF schema, stateless workers).

Each item: **why**, **what**, **files**, **effort** (human / with-CC), **risk**, **priority**.

---

## Track A — More detection (the product's actual job)

The point of a SIEM is catching things. Today: 5 rules, and **3 OCSF classes the parsers can feed
are unused** (`3003` Account Change, `4002` DNS/HTTP, `1001` File System). That's free coverage
left on the table. Rules are YAML — the marginal cost of a rule is minutes, so this is the highest
value-per-effort track.

### A1. Detection coverage matrix (do this first — it's the map)
**Why:** rules today are ad-hoc. Without a coverage map you don't know what you're blind to.
**What:** a matrix of {OCSF class emitted by a parser} × {MITRE ATT&CK tactic} → which rules cover
it, which are gaps. Ground it in what the 6 parsers actually emit (not hypothetical sources). Commit
it as `contracts/detection-coverage.md` and keep it updated per rule PR.
**Effort:** 0.5d / 30min. **Risk:** none (doc). **Priority: P0** — it scopes everything below.

### A2. New rules against unused classes (the free coverage)
Each is one YAML file + one test case in `test_engine_distinct_rules.py`, following the existing
5-rule pattern. Grouped by the class that already has a producer:

**Authentication (3002) — parsers already emit this:**
- `common_password_spray.yml` — inverse of brute-force: one *account* failing auth from *many
  distinct source IPs* in a window (distinct-count on `src_endpoint.ip`, group by `actor.user.name`).
  Catches slow spray that brute-force (per-IP) misses.
- `common_impossible_travel.yml` — same account, successful auth from two geographically distant
  IPs within a window too short to travel. Needs a geo lookup (defer the geo enrichment to A5; ship
  the rule logic keyed on distinct `src_endpoint.ip` country if/when geo lands).
- `common_after_hours_admin.yml` — privileged/admin logon outside a configurable business-hours
  window. Time-of-day predicate — needs A3's richer rule grammar.

**Account Change (3003) — needs the Windows parser to emit 4720/4726/4728/4732 (see A4):**
- `common_priv_grant.yml` — privilege/role grant (`activity_id 5`) — a classic persistence step.
- `common_rapid_account_lifecycle.yml` — account create + privilege-grant + (later) delete by the
  same actor in a short window: the "create a backdoor admin, use it, clean up" pattern.

**Network (4001/4002):**
- `common_dns_exfil.yml` — high volume of distinct DNS queries to one domain (DNS tunneling) — needs
  a DNS parser (A4) emitting `4002`.
- `common_beaconing.yml` — regular-interval connections from one host to one external dst (C2
  beacon). Needs a periodicity primitive (A3) — flag as harder, design before building.

**Data/API (6003/6005) — bank + DC verticals:**
- `bank_mass_card_read.yml` — one account reading many distinct card/PAN records in a window
  (distinct-count) — the PCI headline scenario.
- `dc_privileged_container.yml` — container launched with privileged/hostPath flags via the k8s/
  hypervisor API. Needs a k8s-audit parser (A4).

**Effort:** ~0.5d each / ~10min each with CC once the grammar (A3) and producers (A4) exist.
**Risk:** low per rule; **each needs a producer that emits the field** — the twice-shipped "dormant
rule" bug. A1's matrix + a CI check (A6) prevent that. **Priority: P1** (the auth-class ones ship
now against existing parsers; the rest gate on A4).

### A3. Richer rule grammar (robust logic, unblocks half of A2)
**Why:** current rules do equality-match + count/distinct-count over a window. Real detections need
more: numeric comparison (`>`, `<`), time-of-day, field-not-in-allowlist, and periodicity.
**What, incrementally:**
- Comparison operators in selections (`score: {gt: 60}`) — small evaluator extension.
- Time-of-day / day-of-week predicate (`after_hours_admin`).
- Allowlist/suppression: a rule can reference a named allowlist (`src_endpoint.ip not in
  corp_ranges`) so known-good doesn't alert. **This is the #1 false-positive control** and today
  there's none.
- (Harder, design-first) periodicity/beaconing primitive over the window.
**Files:** `services/ws4-detection/engine.py` (evaluator), `contracts/sigma-convention.md` (document
the grammar), new `contracts/allowlists/`.
**Effort:** 2-3d / half-day. **Risk:** medium — the evaluator is a security-sensitive RCE-free
surface (T4); every extension must stay non-eval and fail-closed (the RecursionError fix set the
precedent). Unit-test each operator adversarially. **Priority: P1.**

### A4. More parsers (more sources = more to detect on)
Priority order by coverage unlocked:
- **Windows Security extended** — 4720/4722/4726/4728/4732/4740 (account create/enable/delete/
  group-add/lockout) → emits `3003` Account Change, unblocks A2's account-lifecycle rules. The
  `windows_eventlog` parser already exists; this is extending its EventID map. **Cheapest high-value
  parser.**
- **DNS / proxy** (Bind/Unbound query logs, Squid) → `4002`, unblocks DNS-exfil/beaconing.
- **Kubernetes audit log** → `6003`, unblocks `dc_privileged_container`.
- **Generic CEF/LEEF** → covers the long tail of appliances (many vendors emit CEF) with one parser.
- **Cloud (CloudTrail/Azure activity)** — the biggest real-world gap; large, defer to v0.4.
**Files:** `services/ws2-normalization/parsers/<name>.py` + registration + test, per the existing
6-parser pattern (`docs/adding-a-parser.md`).
**Effort:** ~0.5-1d each / ~20min each with CC. **Risk:** low (isolated, well-worn pattern). Each
parser processes untrusted input — the ReDoS/DoS discipline from the security sweep applies.
**Priority: P1** (Windows-extended first).

### A5. Enrichment (makes rules smarter) — reputation + geo DONE; asset-criticality deferred
**Why:** rules are blind to context. Geo, threat-intel reputation, and asset-criticality would sharpen
every rule.
**Done:** `services/ws2-normalization/enrichment/` — a WS-2 post-normalize stage (parse → enrich →
validate, wired into `normalize_one`) adding OCSF-additive `src_endpoint.reputation` from a local IOC
list (`contracts/enrichment/ioc.yml`, exact-IP + CIDR longest-prefix) and `src_endpoint.location`
(country) from a local CIDR→country map (`contracts/enrichment/geoip.yml`, a lightweight GeoIP
stand-in with `INTERNAL` tagging). Offline/air-gap-safe (local YAML only), additive (never overwrites
a parser field), fail-open (missing/bad data leaves the event flowing). Enriched events still validate
against Contract A; the fields are added to the OpenSearch event mappings so they're queryable. 12
unit tests. No topic/contract change — enrichment is in-process in WS-2, not a new bus stage.
**Deferred:** (1) **asset-criticality from WS-6** — bus-only coupling means WS-2 can't call WS-6's API
live; would need WS-6 to publish criticality on a topic WS-2 subscribes to, or a local
asset-criticality export. Deferred to keep this pass offline-file-only and avoid a cross-WS coupling
change. (2) **A full GeoIP resolver** (MaxMind etc.) behind the same `src_endpoint.location` field —
the local CIDR map is a deliberate coarse stand-in; the field shape (and any rule reading it) won't
change when a real resolver lands.
**Priority: P2 — reputation+geo closed; the rules that CONSUME this (reputation, impossible-travel)
are A2 follow-ups (adding one now would also perturb the acceptance e2e's alert counts — do it with
the rule work, not the enrichment work).**

### A6. Anti-dormancy CI check (robust logic guardrail)
**Why:** a rule keyed on a field no parser emits has silently shipped **twice**. This class of bug
must be structurally impossible.
**What:** a CI test that, for every rule in `contracts/rules/`, asserts at least one parser emits
every field the rule filters/groups/distinct-counts on (build a field-inventory from the parsers'
sample outputs; fail the build if a rule references a field nothing produces).
**Files:** `tools/check_rule_producers.py` + wire into `run_all_tests.sh` and CI.
**Effort:** 1d / 1h. **Risk:** none. **Priority: P0** — cheap, kills a recurring bug class.

## Track B — Robust logic & architecture (the reach fixes)

From the architecture review. These don't add features; they stop the system breaking at scale or
under load.

### B1. Rule pre-filter — the one real architecture refactor (§4 of the review)
**Why:** detection is O(rules × events) — every event evaluated against every rule. Fine at 5 rules,
quadratic pain at 50+. ARGUS-Sec trains on *thousands* of Sigma rules; if ARGUS ever ingests a
fraction of those, this is the throughput wall.
**What:** index rules by `class_uid` (and where possible `activity_id`) so an event only evaluates
against candidate rules for its class. A dict `{class_uid: [rules]}` built at load; `Detector.process`
looks up `event["class_uid"]` and evaluates only that bucket. Stateful rules keep their windows.
**Files:** `services/ws4-detection/engine.py`, `main.py`.
**Effort:** 2d / half-day. **Risk:** medium — must preserve exact current firing behavior (golden-
fixture regression: every existing rule fires identically before/after). **Priority: P1** — design it
deliberately, it's structural.

### B2. Backpressure policy decision (review R-C) — DONE, using the plan's own recommendation
**Why:** producers `xadd` unbounded; a log flood grows Redis until OOM. But naive `MAXLEN` trimming
drops *unconsumed* events = audit-completeness violation for a bank. This is a product decision, not
a patch.
**Done:** implemented the plan's stated recommendation — **shed at the ingest edge**, never trim
mid-pipeline, so nothing already-accepted is ever lost.
- `services/ws1-collectors/collectors/syslog_udp_server.py`: a `_TokenBucket` (capacity==rate,
  continuous refill) sheds excess datagrams BEFORE `bus.produce()`. UDP is connectionless — there's
  no producer to block — so shedding is the only real lever at this edge. `events_shed` counter +
  a log warning throttled to 1/sec (a flood must not become a logging DoS too). Default
  unlimited; `main.py` applies `SYSLOG_MAX_EVENTS_PER_SEC` (default 2000/s) for real deployments.
- **Zero-loss-under-flood fallback, now built (opt-in):** `services/ws1-collectors/collectors/
  spool.py`'s `BoundedSpool` — a FIFO, byte-capped, disk-backed JSONL queue. A datagram that would
  be shed (rate limit) or dropped (bus produce failed) is written here instead when
  `SYSLOG_SPOOL_PATH` is set; a background thread replays entries back into the bus in order,
  stopping at the first failure to avoid reordering. Still bounded, not infinite — once the spool
  itself hits `SYSLOG_SPOOL_MAX_BYTES` (default 64 MiB), the event is truly lost, but that boundary
  is now an explicit, configurable, distinctly-counted number (`events_lost`) instead of "everything
  over the rate limit, forever." Disabled by default (no behavior change, no disk I/O) — a design
  partner needing audit-completeness under burst traffic opts in via env var.
- `services/shared/bus.py`: `depth(topic)` on both backends (deque len / real XLEN).
- `services/ws1-collectors/main.py`: `_start_depth_watchdog()` background thread, `RAW_EVENTS_DEPTH_WARN`
  (default 100000) — monitoring-only, an operator signal before OOM, NOT an enforcement ceiling; the
  hard cap is the ingest-edge shedding above.
**Files:** as above. `test_syslog_udp.py` covers the token bucket, shedding, and `depth()`.
**Priority: P1 (decision was P0) — closed.** Note: this default (shed-at-edge, no spool) was applied
per the plan's own explicit recommendation, not a fresh ask to the user — revisit if that
assumption turns out wrong for a real deployment.

### B3. Dual-backend behavioral contract test (review R-A)
**Why:** MemoryBus (tests) and RedisBus (prod) have different semantics; redelivery/DLQ is only
partially exercised on MemoryBus. Test-vs-prod fidelity gap.
**What:** run the runner's full ack/redelivery/DLQ/poison-DLQ suite against *both* backends in CI (the
`redis:7` service-container job exists — extend it to the whole behavioral suite, not just the smoke
path).
**Files:** `services/shared/test_runner.py`, `.github/workflows/ci.yml`.
**Effort:** 1d / 1h. **Risk:** none. **Priority: P1.**

### B4. Rule hot-reload + validation gate — validation gate DONE; hot-reload deferred
**Why:** today rules load at startup; adding a rule = restart. And a malformed community rule can
only be caught by reading it.
**Done (validation gate):** `tools/validate_rules.py` + `tools/test_validate_rules.py`, wired into
`run_all_tests.sh`/CI. Validates schema (UUID id, level enum, score bounds, stateful pairing), that
the `condition` parses under the REAL T4 evaluator and references only defined selections, that every
operator is one the engine implements (unknown operators rejected at gate time, not silently
fail-closed at runtime), that `not_in`/`outside_hours` references are well-formed and exist, and that
ids are unique. Reuses the engine's own parser/operator set so the gate can't drift from runtime.
Complements A6's `check_rule_producers.py` (dormancy). Documented in `CONTRIBUTING.md`. 20 unit tests,
each check has an adversarial reject case.
**Deferred (hot-reload):** the SIGHUP-reload-in-WS-4 half is not built. Lower value than the gate
(the gate makes contributing SAFE; hot-reload only saves a restart) and it needs care around
re-loading rules mid-consume without dropping in-flight window state. Pick up in a later pass if
operators ask for it.
**Priority: P2 — gate closed, hot-reload deferred.**

### B5. HA design (review R-B) — design only in v0.3
**Why:** Redis + OpenSearch single points of failure; correct for local/air-gapped, unacceptable for
central.
**What:** a design doc (not code) for Redis Sentinel/Cluster + OpenSearch replicas for the central
tier. Decide, don't build, in v0.3.
**Effort:** 1d design. **Priority: P3** (gate on the central tier actually being built).

## Track C — Better design (the analyst experience)

The dashboard renders tables. A SIEM lives or dies on the analyst's triage loop.

### C1. Triage workflow, not a table
**Why:** the "millions of signals → handful of decisions" promise is a *workflow*, not a list. Today
an analyst sees alert rows with no way to act.
**What:** an alert detail/triage view — the alert, its triggering events, the WS-5 AI verdict +
rationale, the involved asset (from WS-6), and actions (acknowledge, assign, mark FP/TP, suppress).
The FP/TP feedback is also the data ARGUS-Sec needs (closes a loop). Escalation state (`new` /
`triaged` / `closed`) persisted.
**Files:** `services/ws7-dashboard/` (needs a small state API — either extend WS-6 or a new tiny WS).
**Effort:** 1-2wk / 2-3d. **Risk:** medium (new persistence). **Priority: P1** — this is the single
biggest product-value item in the whole plan.

### C2. Live updates + real IA
**Why:** the dashboard polls; alerts should stream. And the 3-level nav (device/inventory/sources) is
mock-era IA, not a triage IA.
**What:** WebSocket/SSE push for new alerts; re-architect nav around the analyst's job (Triage queue →
Investigation → Coverage/health). Show the LIVE vs. mock indicator that's tracked but unused today.
**Effort:** 1wk / 2d. **Risk:** low. **Priority: P2.**

### C3. Detection coverage view
**Why:** operators need to see what they're *not* catching (surfacing A1's matrix in the UI).
**What:** a MITRE ATT&CK-style coverage heatmap: which tactics/techniques have rules, which are gaps,
sourced from A1.
**Effort:** 3d / half-day. **Priority: P3.**

## Sequencing (what to actually do, in order)

1. **P0, cheap, unblocks everything:** A1 (coverage matrix), A6 (anti-dormancy CI), the B2 *decision*.
2. **P1, high value:** A4-Windows-extended + A2 auth-class rules (ship coverage now), A3 (grammar +
   allowlists — kills false positives), B1 (rule pre-filter — the scale unblock), C1 (triage
   workflow — the product-value headline), B3 (dual-backend test).
3. **P2:** rest of A2/A4 parsers, A5 enrichment, B4 hot-reload, C2 live UI.
4. **P3 / design-only:** B5 HA, C3 coverage view.

**Golden rule for all of it (from the architecture review):** don't disturb the keystone. Bus-only
coupling, OCSF-additive fields (never a hard dependency on a new field — tolerant readers), stateless
workers, RCE-free fail-closed evaluator. Every item above is additive to that spine, not a rewrite of
it.
