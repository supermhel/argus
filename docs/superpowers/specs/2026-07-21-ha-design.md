# B5: central-tier HA design (2026-07-21)

**Design only, per the v0.3 plan's own scope decision for this item — no
code in this doc.** Decide the shape of a highly-available central-tier
deployment; build it later, gated on the central tier actually being staffed
and needed (the v0.3 plan's own P3 priority call, unchanged here).

## Scope

FENGARDE's default deployment (`infra/docker-compose.yml`) is single-node:
one Redis, one OpenSearch, one replica of each service. That is correct for
local/air-gapped/edge deployments (the repo's primary target per README).
This doc is for the "central tier" case: an MSP or larger org running this
as shared infrastructure across many tenants, where a single Redis or
OpenSearch node is an unacceptable single point of failure.

## What's already HA-ready, unchanged

These exist today and need zero new work to run multi-replica:

- **WS-4 detection window counters.** `RedisWindowCounter` (`services/
  ws4-detection/window.py`) already keeps sliding-window state in Redis
  sorted sets, not per-process memory — this was T6's whole point (the Opus
  review finding that a per-replica deque splits event counts and a
  brute-force alert never reaches threshold under horizontal scaling). Every
  WS-4 replica pointed at the same Redis sees the same global count. No
  change needed to run N replicas of WS-4.
- **Bus consumption.** `services/shared/bus.py`'s `_RedisBus` already uses
  real Redis Streams consumer groups (`XGROUP`/`XREADGROUP`/`XACK`/
  `XCLAIM`/`XAUTOCLAIM`) — the whole point of a consumer group is that
  multiple readers in the same group split the stream's messages between
  them, each message delivered to exactly one live consumer, with automatic
  redelivery (`claim_pending`) if a consumer dies mid-processing. `make
  chaos` (2026-07-18, `scenarios=40 lost=0 duplicated=0`) already proved
  this holds under mid-replay SIGKILL of every pipeline stage. Running N
  replicas of any WS is already safe today.
- **Alert idempotency.** `Rule.alert_key()` (T7, deterministic id) means two
  replicas processing the same redelivered event produce the SAME alert_id
  — OpenSearch's index-by-id semantics dedupe it. No coordination needed
  between replicas beyond what the bus + deterministic id already give.
- **Sessions (v0.5).** `services/shared/sessions.py`'s `RedisSessionStore`
  (added this session, see SSOT.md) means RBAC login state is no longer
  tied to one WS-3 process — any replica can resolve any session's token
  once `FENGARDE_SESSION_BACKEND=redis` is set.
- **OpenSearch writes.** WS-3's CAS path (`find_alert_versioned`/
  `index_cas`, `_seq_no`/`_primary_term` optimistic concurrency) already
  handles concurrent writers to the same alert document from multiple WS-3
  replicas — proven live against a real cluster (SSOT.md).

**Conclusion so far: the application layer's multi-replica story is
already mostly solved.** What's left is the two stateful STORES
themselves (Redis, OpenSearch) each being a single node.

## What's NOT solved: single-node Redis and single-node OpenSearch

### Redis: recommend Sentinel, not Cluster

**Decision: Redis Sentinel (primary/replica + automatic failover), not Redis
Cluster (sharded).**

Rationale:
- FENGARDE's Redis usage is entirely single-key-space per topic (Streams,
  sorted sets, session hashes) — nothing here needs Cluster's horizontal
  sharding for throughput; a single primary can hold the whole working set
  for any realistic MSP tenant count based on the measured EPS numbers
  (README's `fengarde_bench.py` results).
  Cluster's payoff is write-throughput sharding, which isn't the bottleneck
  here; its cost is real — Cluster forbids multi-key operations across
  hash slots, and `RedisWindowCounter`'s pipelined ZADD+ZREMRANGEBYSCORE+
  ZCARD+EXPIRE on one key is fine under Cluster (single-key pipeline), but a
  future feature needing a multi-key transaction would have to work around
  slot placement. Sentinel avoids that constraint entirely for a workload
  that doesn't need sharding.
- Sentinel gives automatic primary failover (a replica promotes when the
  primary is unreachable) with the existing `redis-py` client's built-in
  Sentinel support (`redis.sentinel.Sentinel`), a small, well-understood
  change to `services/shared/bus.py`'s connection construction — swap
  `redis.Redis.from_url(REDIS_URL)` for a Sentinel-aware connection when
  `REDIS_SENTINEL_HOSTS` (or similar) is set, same "opt-in, unset = today's
  behavior" convention every other v0.4/v0.5 feature in this repo follows.
- **Failure mode during failover**: a brief write-unavailable window while
  Sentinel elects a new primary (typically single-digit seconds). Bus
  producers/consumers already retry-with-backoff on transient errors
  (existing `_RedisBus` behavior); this is a availability blip, not a
  correctness risk, because nothing here assumes Redis is always reachable
  — WS-4's own `set_counter` fallback to a per-process deque on Redis
  construction failure (main.py) already demonstrates the "degrade, don't
  crash" posture, though a mid-run Sentinel failover would need the retry
  path (not the one-time construction fallback) to actually recover.

### OpenSearch: replica shards + a real cluster, not a design change

**Decision: standard OpenSearch multi-node cluster with replica shards
(`number_of_replicas: 1` or higher), no code changes needed in this repo.**

- Every index template already sets `number_of_replicas: 1` (see any
  `contracts/opensearch-mappings/*.json`) — that setting is already
  cluster-ready; on a single-node dev cluster it just sits unassigned
  (yellow health, which is exactly what local `make up` shows today,
  confirmed during this session's live verification). Pointing
  `OPENSEARCH_URL` at a real multi-node cluster with those same templates
  installed (`tools/migrate_opensearch.py`, proven live this session) makes
  the replica shards actually assign — zero application-code change.
- WS-3's CAS path is already cluster-safe (proven this session against a
  real cluster's `_seq_no`/`_primary_term`), so replica promotion during a
  node failure doesn't introduce a new write-conflict class beyond what CAS
  already handles for concurrent single-node writers.
- ISM policies (this session's ILM→ISM fix) are cluster-wide, not
  per-node, so retention keeps working the same way across a multi-node
  cluster.

## What stays single-writer (accepted, not a gap)

- **WS-6 inventory** (SQLite-backed). A real central-tier deployment would
  need this migrated to a real database (Postgres) or run behind a single
  writer with read replicas — out of scope for this doc; flagged as a
  known future item, not silently assumed solved.
- **Dashboard nginx**. Stateless (all state lives in WS-3/WS-6/OpenSearch
  behind it), so N replicas behind a load balancer work today with zero
  change — "single-writer" doesn't even apply here, noted for completeness.
- **RBAC session store, if `FENGARDE_SESSION_BACKEND` stays `memory`**.
  The default remains single-process by design (this session's SSOT entry)
  — this is the ONE piece that needs an explicit opt-in
  (`FENGARDE_SESSION_BACKEND=redis`) to become multi-replica-safe; without
  it, RBAC-enabled deployments must stick to one WS-3 replica or accept
  that a login on one replica isn't visible to another.

## Summary table

| Component | Multi-replica-safe today? | What's needed for full central-tier HA |
|---|---|---|
| WS-1/2/4/5/7 (stateless or Redis-window-backed) | Yes | Nothing |
| WS-3 (OpenSearch CAS writes) | Yes | Nothing |
| Bus (Redis Streams consumer groups) | Yes (single Redis node) | Redis Sentinel for node failover |
| Detection window counters | Yes (single Redis node) | Same Sentinel dependency |
| RBAC sessions | Opt-in (`FENGARDE_SESSION_BACKEND=redis`) | Operator sets the env var; still single Redis node underneath |
| OpenSearch storage | Yes (templates already set replicas) | A real multi-node cluster; zero app-code change |
| WS-6 inventory (SQLite) | No | Out of scope for this pass — flagged, not solved |

## Non-goals of this pass

No code changes. No Sentinel/Cluster deployment manifests written. No
Postgres migration for WS-6. This is the decision record the v0.3 plan
asked for (B5: "decide, don't build, in v0.3" — now v0.5) plus an honest
accounting of how much of the multi-replica story later work (T6's
RedisWindowCounter, this session's RedisSessionStore, the live-verified
OpenSearch CAS/ISM/migrate work) already solved without anyone setting out
to solve "HA" as its own project.
