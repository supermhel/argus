### (historical) reopened by Opus 4.8 final review

- **Direction (still defensible): a shared service runner** (`services/shared/runner.py`).
  But the decision was locked prematurely — the Opus review proved the doc was
  architecture-by-assertion. Three load-bearing claims are FALSE or unwritten against
  the actual `bus.py`:
  1. **"consume() blocks on Redis" is false.** `bus.py:70` returns on the first empty
     `xreadgroup` (`if not resp: return`). One thread over N topics STARVES topics 2..N
     under load on topic 1 — the exact problem the doc claimed to solve. The runner needs
     **N+1 threads** (per-topic + health), unmentioned. Real T0 cost, not "one-line mains."
  2. **Redelivery doesn't exist.** `bus.py:69` reads `{topic: ">"}` = new messages only.
     Unacked messages sit in the Redis PEL and are NEVER redelivered. T7's "handler throws
     → redelivered" requires `XAUTOCLAIM`/`XPENDING`; the cap count must come from
     `XPENDING.times_delivered` (survives restart). None written. Untestable on MemoryBus
     (ack is a no-op, `bus.py:46-48`), so the "validate both backends" criterion is
     Redis-only for the feature that matters.
  3. **consume()'s return-on-empty is incompatible with a daemon** — the runner must wrap
     it in its own `while not shutdown` re-entry loop; "consume() unchanged" is wrong.

- **Status: T3 must do a REAL spike that writes the actual loop BEFORE T0 lands.** Prove
  (i) N-topic non-starvation (threads or async), (ii) redelivery via `XAUTOCLAIM` +
  `XPENDING.times_delivered`, (iii) the re-entry wrapper. Then amend the T3 doc with the
  real mechanism, or fall back to per-service loops. The "collapses T0+T2+T7" claim is
  optimistic until this exists.
